# نظام نقاط البيع والكاشير (Point of Sale - POS)

هذا الملف (Skill) يوفر معايير هندسية متكاملة لبناء وتوسيع نظام نقاط البيع (POS) داخل التطبيق، معتمداً على بيئة العمل الحالية (React + Dexie.js) لضمان العمل بدون اتصال بالإنترنت (Offline-First).

أنت وكيل برمجي محترف. المطلوب بناء شاشة نقاط بيع (POS) متكاملة وحديثة، مصممة خصيصاً للسرعة وتتحمل ضغط العمل العالي (High-Volume Transactions)، مع دعم شاشات اللمس (Touch-friendly) وقارئات الباركود (Barcode Scanners)، وارتباطها الوثيق بالمخزون والمحاسبة والورديات.

## أولاً: أقسام وتطبيقات النظام الشاملة (Comprehensive Modules)
يوفر النظام الوحدات التالية:
1. **واجهة الكاشير السريعة (POS Terminal)**: نقطة البيع للمس والباركود وإنهاء الفواتير بسرعة.
2. **إدارة الورديات والصندوق (Shift & Drawer Management)**: تسليم الدرج والزيادة أو العجز.
3. **تعليق واسترجاع الطلبات (Hold & Resume)**: تعليق فاتورة لخدمة عميل آخر.
4. **تعدد طرق الدفع (Split Payments)**: كاش، بطاقة، شبكة أو استخدام الولاء.

## ثانياً: المعايير التقنية والتصميمية (Core Guidelines)

### 1. الأثر المالي والقيود المحاسبية (Global Financial Impact) - **مهم جداً**
- **الفاتورة النقدية و الخزينة**: الخصم اللحظي. بمجرد دفع الفاتورة، يُعد قيد آلي مباشر `journalEntries` يدخل المبلغ لحساب "الخزينة" أو "البنك" ومقابله حساب "المبيعات".
- **تكلفة المبيعات (COGS)**: الفاتورة ذاتها تكون مصحوبة بقيد ترحيل يخصم كلفة البضاعة من "مخزون" ويزيدها لـ "تكلفة المبيعات".
- **تسوية الوردية (Z-Report)**: تقفيل الكاشير آخر اليوم يوضح الفروقات (عجز يُسجل على الكاشير، زيادة تُسجل إيرادات متفرقة).

### 2. الهيكل التقني والقيود (Core Architecture)
- **البيانات المحلية (Offline-First)**: جميع البيانات يجب أن تُخزن محلياً باستخدام `IndexedDB` عبر مكتبة `Dexie.js`. هذا النظام مصمم ليعمل في السوبر ماركت أو الصيدليات، لذا يجب ألا يتوقف البيع مهما حدث للاتصال.
- **التكامل المباشر (Real-time Deductions)**: في نقطة البيع لا يوجد خطوات متعددة (أمر بيع ثم تسليم)، الفاتورة بمجرد دفعها تقوم بخصم المخزون وتسجيل النقد في درج الكاشير آلياً، وتوليد قيد من ح/الخزينة و ح/تكلفة المبيعات، إلى ح/المبيعات و ح/المخزون.
- **إدارة الـ Focus والباركود**: النظام يجب أن يستقبل إدخالات قارئ الباركود (الذي يتصرف كلوحة مفاتيح تكتب كوداً متبوعاً بـ Enter) بسرعة فائقة وبدون تشتيت الشاشة.

### 2. هيكل قاعدة البيانات المقترح للـ POS (Dexie Models)
```typescript
// فاتورة نقطة البيع (POS Receipt/Invoice)
export interface POSInvoice {
  id?: number;
  receiptNumber: string; // متسلسل سريع
  date: string;
  shiftId: number; // رقم وردية الكاشير
  cashierId: number;
  customerId?: number; // يمكن أن تكون للعميل الافتراضي Cash Customer
  totalAmount: number;
  taxAmount: number;
  discountAmount: number;
  netAmount: number;
  paidAmount: number;
  changeAmount: number; // المتبقي للعميل
  paymentMethod: 'Cash' | 'Card' | 'Split'; // دعم الدفع المزدوج
  status: 'Completed' | 'Refunded' | 'Hold';
}

// السطور والمشتريات داخل الإيصال (Receipt Lines)
export interface POSInvoiceLine {
  id?: number;
  posInvoiceId: number;
  itemId: number;
  quantity: number;
  unitPrice: number;
  total: number;
}

// الوردية وأدراج الكاشير (Shifts & Cash Drawers)
export interface CashierShift {
  id?: number;
  cashierId: number;
  deviceId: string; // معرف نقطة البيع Terminal
  openingDate: string;
  closingDate?: string;
  openingCash: number; // عهدة الدرج عند الفتح
  expectedCash?: number; // ما يحسبه النظام
  actualCash?: number; // ما قام بعده الكاشير
  cashDifference?: number; // العجز أو الزيادة
  status: 'Open' | 'Closed';
}
```

## ثانياً: المتطلبات الوظيفية المفصلة للنظام (Functional Requirements)

========================
1. واجهة الكاشير السريعة (The POS Terminal)
========================
- **تصميم الشاشة**: يجب أن لا تحتوي على أشرطة تمرير معقدة.
  - الجزء الأيمن أو الأيسر يشمل شبكة المنتجات (Product Grid) بالصور لمسية للبيئات مثل المطاعم والمقاهي.
  - الجزء الآخر يمثل تذكرة الـ Receipt الحالية.
  - جزء سفلي أو جانبي للوحة المفاتيح الرقمية (Numpad) والدفع.
- **الإدخال السريع**: حقل إدخال واحد للباركود يكون فيه الـ Focus النشط دائماً. عند إدخال الباركود يضاف الصنف للفاتورة تلقائياً بدون تدخل الفأرة.
- **تكرار الأصناف**: تمرير نفس الباركود يزيد الكمية (+1) ولا يضيف سطراً جديداً لتقليل الزحام وورق الطابعة.

========================
2. إدارة الورديات وأدراج النقدية (Shift Management)
========================
- **استلام الدرج (Shift Open)**: عند بدء اليوم، لا يستطيع الكاشير البيع إلا بعد فتح وردية وتحديد كم معه كاش مبدئي (العهدة/Float).
- **تصريحات النقدية ومسحوبات الدرج**: تسجيل مدفوعات صغيرة من الدرج (مصروف نظافة، شراء بضاعة سريعة) كعملية داخل الوردية.
- **تسليم الدرج (Shift Close / Z-Report)**: نهاية يوم العامل يقوم بإعطاء التقرير (X للطباعة خلال اليوم) وإغلاق اليوم (Z لإنهاء الوردية). يطابق المتوفر معه بالدرج مع ما توقعه النظام، والعجز يسجل عليه أو كخسارة، والزيادة تسجل أرباح متفرقة.

========================
3. العمليات الأساسية المتقدمة للكاشير (Advanced POS Ops)
========================
- **تعليق الطلب (Hold/Resume Bill)**: إذا نسي العميل محفظته وأراد الكاشير خدمة العميل التالي، يمكنه تعليق الفاتورة والعودة لها بعد دقائق.
- **إلغاء الصنف وإلغاء الفاتورة (Void Item & Cancel Bill)**: أزرار واضحة لحذف صنف من القائمة، ويفضل في الأنظمة الأمنِة أن تتطلب كلمة سر المدير للحد من السرقة التدريجية.
- **تطبيق الخصم (Discount)**: خصم مبلغ أو نسبة مئوية (محمي بالصلاحيات).
- **المرتجعات (Returns)**: السماح بإرجاع فاتورة وخصمها من خزينة وردية الكاشير، مع إعادة المخزون.

========================
4. طرق ووسائل الدفع (Payments Processing)
========================
- دفع نقدي (Cash).
- دفع ببطاقة مدى أو ائتمان (Card/Banka Terminal).
- **الدفع المنفصل (Split Payment)**: مثلاً دفع الفاتورة 100 ريال كالتالي: 50 كاش و 50 عبر البطاقة. النظام يجب أن يتعامل معها بمرونة وتوجيه الأموال للحسابين الماليين المنفصلين.
- حساب الباقي (Change Calculation) سريع بمجرد إدخال الملبغ المستلم من العميل.

========================
5. البحث والفئات (Categories & Search)
========================
- تصنيف المنتجات في فئات (Tabs) بحيث يتفادى الكاشير الاعتماد الكلي على الباركود للأصناف التي ليس عليها باركود (مثل الخضار أو المخبوزات).
- إمكانية البحث النصي في الأصناف.

========================
6. ترابط الـ POS بالمحاسبة والمخزون
========================
- القيود تتولد عند بيع كل فاتورة، أو يتم تجميع مبيعات الوردية في قيد واحد كبير (مبيعات مجمعة) عند إغلاق הوردية لتقليل إزعاج دفتر اليومية، وهذا الخيار مفضل لأنظمة التجزئة.
- تتبع مخزون فوري، ومنع بيع أصناف كمياتها صفر (يمكن تفعيل تخطي هذا كإعداد للسماح بالسالب).

========================
7. الطباعة والإيصالات (Printing)
========================
- تنسيق إيصال بحجم 80mm متوافق مع الطابعات الحرارية عبر الويب.
- يحتوي على: اللوجو متمركز፣ بيانات الشركة والرقم الضريبي، رقم الوردية والكاشير، وقت الطباعة، الأصناف، الإجمالي، الباركود أو الـ QR code الخاص بالضريبة للجهة المختصة.

========================
8. اختبارات القبول (Acceptance Criteria)
========================
بعد التنفيذ يجب التأكد من:
1. عدم قدرة الكاشير على الدخول لشاشة البيع إلا إذا كان لديه وردية مفتوحة.
2. استخدام ماسح الباركود يضيف الصنف فوراً، ويعرض السعر الإجمالي بشكل لحظي.
3. معالجة فاتورة مقسمة الدفع والتأكد أن الصندوق الكاش زاد بالقسم النقدي، وحساب البنك زاد بالبطاقة.
4. إغلاق وردية بها مبيعات ومصروفات نقدية، وإظهار مطابقة النقدية مع ما يتوقعه النظام وعرض الإنحراف.
5. تعليق طلب وإنشاء طلب جديد للعميل التالي، ثم استدعاء الطلب المعلق وإنهائه.
